### 对象同步
* 用于简化对客户端的数据同步工作
* 比如 玩家对象上的item 当物品有增删改 服务器一般都要主动推送给客户端
* 一个改动就要写一条push协议 所以push协议非常多 
* 自动的对象同步机制 自动监听数据变动并直接向客户端发送这些变动
* 设计目标 服务器业务逻辑层面尽可能少写push协议

### 内部服务与客户端对象同步的区别
* 内网网速好 也没流量费用 所以我们可以直接打包大对象 比如物品有变动 我们可以直接把物品全部对象打包 全量同步到另一个服务上
* 但外网 我们不能这么做 流量会爆炸 这个也是不得不采用push协议的原因

### 想法来源
* 来自于畅游的一位主程 想法不错 我对他的方法进行了改进
* 尽量不破坏原生语义

### 难点
* lua的对象就像树的根节点和子节点的关系
* 如果子节点变动如何溯源 一层一层找到根节点
* 还有就是一些覆盖性的操作
```
items[100].id=100 items[100]=nil
如果一个对象了数据 然后又删了 然后又加了 会比较麻烦
```
* 每秒把有变动的玩家都推给客户端 所以还需要有一个协议去兼容
* 那位主程的方法是自定义协议 如果客户端不用lua就会很麻烦

### 实现 具体实现看代码 总共也就100行左右

```
最早期用的是metatable 怎么做都做不干净
obj = {__INFO={id=10}}
setmetatable(obj, {
    __index = function(tb, k) return obj.__INFO[id] end

最早期的实现大概都是类似的做法 对象里面挂一个子对象 真实的对象信息放在__INFO的子对象里
然后每次读或者写都会出发__index __newindex的元方法
缺点 既费内存又费cpu 尤其小对象一多 大概能多用50%的内存 也大大加大了gc的负担
```

```
player.role.level = 10
player.get("role").set("level", 10)
费cpu是一定了 但是没有多余的内存使用
```

```
因为删除毕竟是少量操作 没必要做的太复杂

items[100]={id=100} items[100]=nil

难点就是到时候要遍历这个表 把删除操作分离出来

最后这种改动基本可以兼容sproto protobuf3协议 和客户端解耦
```
